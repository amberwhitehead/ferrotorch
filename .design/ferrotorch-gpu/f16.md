## Summary

`ferrotorch-gpu/src/f16.rs` owns the IEEE binary16 (`half::f16`) GPU
compute kernels. Each kernel is a hand-written PTX `&'static str`
targeting sm_53+ (the architecture floor for the native `cvt.f32.f16`
instruction), loaded via `crate::module_cache::get_or_compile`. f16
buffers are stored as `CudaSlice<u16>` (cudarc `DeviceRepr` for `u16`);
the `backend_impl::CudaBackendImpl` handle is tagged `DType::F16` to
disambiguate it from bf16 (also u16-backed). Mirrors PyTorch's
`aten/src/ATen/native/cuda/BinaryMulKernel.cu`,
`Activation*Kernel.cu`, and `c10::Half` numeric type across
`aten/src/ATen/native/cuda/`.

## Requirements

- REQ-1: Elementwise binary arithmetic — `gpu_mul_f16`, `gpu_add_f16`,
  `gpu_sub_f16`, `gpu_div_f16` — each operating on two
  `CudaSlice<u16>` f16 buffers, using PTX `cvt.f32.f16` to decode to
  f32, then `cvt.rn.f16.f32` to re-encode (round-to-nearest-even,
  native hardware instructions; faster and more correct than bf16's
  software bias).

- REQ-2: Elementwise unary activations — `gpu_silu_f16`, `gpu_relu_f16`,
  `gpu_gelu_f16`, `gpu_exp_f16`, `gpu_log_f16`, `gpu_tanh_f16`,
  `gpu_sigmoid_f16`, `gpu_sqrt_f16`, `gpu_neg_f16` — one PTX kernel
  per op, decode-compute-encode pattern.

- REQ-3: Scalar elementwise — `gpu_scale_f16` (multiply an f16 buffer
  by an f32 scalar parameter).

- REQ-4: Broadcast elementwise — `gpu_broadcast_{add,sub,mul,div}_f16` —
  each takes both inputs' shapes and the output shape, dispatching a
  stride-aware PTX kernel that mirrors PyTorch's broadcasting rules.

- REQ-5: Reduction kernels — `gpu_sum_f16`, `gpu_mean_f16`,
  `gpu_sum_axis_f16`, `gpu_mean_axis_f16` — accumulate in f32
  (PyTorch parity per `c10::accum_type<at::Half>`) and re-encode the
  final result to f16.

- REQ-6: Normalisation primitives — `gpu_layernorm_f16`,
  `gpu_rmsnorm_f16`, `gpu_softmax_f16` — composite kernels combining
  reduce + elementwise apply in a single PTX entry.

- REQ-7: Every `unsafe { ... }` block carries a SAFETY comment naming
  the kernel entry, the buffer-allocation site, the exclusive-mutable-
  borrow invariant, the bound-check guarantee, and the `n as u32`
  non-truncation argument (matching R-CODE-1 leaf-primitive policy).

- REQ-8: `n == 0` is an early-return path that allocates a 0-length
  output (no kernel launch). Mirrors PyTorch's empty-tensor short
  circuit in TensorIterator.

## Architecture

### f16 vs bf16 — the native-PTX advantage

f16 is a true IEEE-754 binary16 (1 sign / 5 exponent / 10 mantissa),
while bf16 is the top 16 bits of an f32 (1 / 8 / 7). Because f16 is a
genuine IEEE format, PTX provides **native** conversion instructions
that the bf16 module has to hand-roll:

- **f16 → f32**: `cvt.f32.f16 %f, %h` (lossless, sm_53+).
- **f32 → f16, round-to-nearest-even**: `cvt.rn.f16.f32 %h, %f`
  (faster and more correct than bf16's `0x7FFF + bit[16]` software bias).

All arithmetic happens in `.f32` registers per thread; storage is
always `u16` (`.b16`) in global memory. No whole-tensor f32 intermediate
materialisation. Reductions accumulate in f32 (PyTorch parity for
half-precision reductions per `c10::accum_type`).

### Elementwise binary (REQ-1)

`pub fn gpu_mul_f16 in f16.rs` is the reference. Each binary kernel
reads `a[i]`, `b[i]` (one `.b16` each), converts to f32 with
`cvt.f32.f16`, computes the math in `.f32`, then re-encodes with
`cvt.rn.f16.f32`. Output stored at `out[i]` as `.b16`. Bound check
matches the bf16 idiom. Wrapper at the Rust level uploads the PTX once
via `module_cache::get_or_compile`, allocs an `n`-element u16 output,
and dispatches via `stream.launch_builder().launch(cfg)`. Non-test
production consumer: `crate::backend_impl::CudaBackendImpl`'s f16 dtype
arm calls the matching `gpu_*_f16` function (e.g.
`gpu_add_f16 in backend_impl.rs` invokes `gpu_add_f16`, `gpu_add_f16 in backend_impl.rs` `gpu_mul_f16`,
`gpu_sub_f16 in backend_impl.rs` `gpu_sub_f16`, `gpu_sub_f16 in backend_impl.rs` `gpu_div_f16`).

### Elementwise unary (REQ-2)

`pub fn gpu_silu_f16/gpu_relu_f16/gpu_gelu_f16 in f16.rs` plus the
transcendental family (`exp/log/tanh/sigmoid/sqrt`) and `gpu_neg_f16`
follow the same decode-compute-encode pattern. `gpu_gelu_f16` uses the
tanh approximation (PyTorch's default `approximate='tanh'` path on
`c10::Half`); the kernel takes `sm_53` because the helper `abs.f32` on
the half path requires it. Non-test consumer: `backend_impl.rs`
(`gpu_neg_f16`), `gpu_neg_f16 in backend_impl.rs` (the activation + transcendental block).

### Scalar / broadcast (REQ-3, REQ-4)

`pub fn gpu_scale_f16 in f16.rs` multiplies the buffer by an f32
scalar. Non-test consumer: `backend_impl.rs`. The broadcast
family (`gpu_broadcast_{add,sub,mul,div}_f16`) takes both inputs'
shapes and the output shape and dispatches a stride-aware kernel.
Non-test consumer: `backend_impl.rs`.

### Reductions (REQ-5)

`pub fn gpu_sum_f16/gpu_mean_f16 in f16.rs` are global; `sum_axis_f16/
mean_axis_f16` are axis reductions. All accumulate in f32 (PyTorch
parity for half-precision reductions; cf. `c10::accum_type<at::Half>` =
`float`). Final result re-encoded to f16. Non-test consumer:
`backend_impl.rs`.

### Normalisation (REQ-6)

`pub fn gpu_layernorm_f16/gpu_rmsnorm_f16/gpu_softmax_f16 in f16.rs`
are composite kernels (mean + variance/RMS + apply in one PTX entry).
`gpu_softmax_f16` uses the max-subtract trick mirroring PyTorch's
`SoftMax.cu::cunn_SoftMaxForward`. Non-test consumer:
`backend_impl.rs` (softmax), `backend_impl.rs` (layernorm), `backend_impl.rs`
(rmsnorm).

### SAFETY discipline (REQ-7)

Every `unsafe { stream.launch_builder(&f)...launch(cfg)? }` is
preceded by a multi-line SAFETY comment matching the bf16 pattern:
compile site, entry signature, buffer alloc, bound check, `n as u32`
non-truncation. R-CODE-1 grandfathers raw CUDA kernel launches.

### n == 0 (REQ-8)

Every public entry point opens with `if n == 0 { return
Ok(stream.alloc_zeros::<u16>(0)?); }` — no PTX launch.

## Parity contract

`parity_ops = []` for this route (S5: INFRASTRUCTURE). Per-op parity
is enforced at the ferrotorch-core op layer; this file is the
dtype-specific implementation invoked when input dtype is `DType::F16`.

Edge cases preserved:

- **NaN propagation**: f32 arithmetic preserves NaN; `cvt.rn.f16.f32`
  preserves NaN bit pattern (cf. IEEE-754 §6.2).
- **±Inf**: `exp(big) → +Inf` (f32), encoded as f16 bit pattern
  `0x7C00`. `log(0)` → `-Inf` (f32 `lg2.approx.f32` on 0 yields
  `-Inf`).
- **Subnormals**: f16 subnormals occur with biased exponent 0;
  `cvt.rn.f16.f32` produces them; `cvt.f32.f16` widens them losslessly.
- **Empty input** (`n == 0`): early return without launching.

## Verification

The integration tests in `ferrotorch-gpu/tests/conformance_gpu_kernels.rs`
exercise the f16 path through the backend dispatch. The
`backend_impl::tests` block (the bf16/f16 dtype arm tests) further
exercises both half-precision codepaths.

Smoke command:

```bash
cargo test -p ferrotorch-gpu --features cuda --lib f16 2>&1 | tail -3
```

Expected: `test result: ok` on a host with a CUDA device. The
`#![cfg(feature = "cuda")]` gate excludes the module on no-CUDA builds.
