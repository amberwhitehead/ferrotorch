## Summary

`ferrotorch-gpu/src/bf16.rs` owns the bfloat16 GPU compute kernels. Each
kernel is a hand-written PTX `&'static str` (no CUDA C++, no nvrtc,
no external toolchain at load time) loaded via
`crate::module_cache::get_or_compile`. bf16 buffers are stored as
`CudaSlice<u16>` (cudarc `DeviceRepr` for `u16`); the
[`backend_impl::CudaBackendImpl`] handle is tagged `DType::BF16` so a
bf16 buffer (2 bytes per element) is never read as f16 or i16. This
mirrors PyTorch's `aten/src/ATen/native/cuda/BinaryMulKernel.cu`,
`BinaryDivTrueKernel.cu`, layered activation kernels, and the
`c10::BFloat16` numeric type used throughout `aten/src/ATen/native/cuda/`.

## Requirements

- REQ-1: Elementwise binary arithmetic — `gpu_mul_bf16`, `gpu_add_bf16`,
  `gpu_sub_bf16`, `gpu_div_bf16` — each operating on two
  `CudaSlice<u16>` bf16 buffers, decoding both inputs to f32 in PTX
  registers, computing in f32, then re-encoding to bf16 with
  round-to-nearest-even via the `0x7FFF + bit[16]` bias trick.

- REQ-2: Elementwise unary activations — `gpu_silu_bf16`,
  `gpu_relu_bf16`, `gpu_fatrelu_bf16`, `gpu_gelu_bf16`, `gpu_exp_bf16`,
  `gpu_log_bf16`, `gpu_tanh_bf16`, `gpu_sigmoid_bf16`, `gpu_neg_bf16` —
  one PTX kernel per op, decode-compute-encode pattern.

- REQ-3: Scalar elementwise — `gpu_scale_bf16` (multiply a bf16 buffer
  by an f32 scalar parameter).

- REQ-4: Broadcast elementwise — `gpu_broadcast_add_bf16`,
  `gpu_broadcast_sub_bf16`, `gpu_broadcast_mul_bf16`,
  `gpu_broadcast_div_bf16` — each takes both inputs' shapes and the
  output shape, dispatching a PTX kernel with stride-aware indexing
  to mirror PyTorch's broadcasting rules.

- REQ-5: Reduction kernels — `gpu_sum_bf16`, `gpu_mean_bf16`,
  `gpu_sum_axis_bf16_bf16`, `gpu_mean_axis_bf16_bf16` — accumulate in
  f32 (PyTorch parity for half-precision reductions) and re-encode
  the final result to bf16.

- REQ-6: Normalisation primitives — `gpu_layernorm_bf16`,
  `gpu_rmsnorm_bf16`, `gpu_softmax_bf16` — composite kernels combining
  reduce + elementwise apply in a single PTX entry.

- REQ-7: Transformer primitives — `gpu_embedding_gather_bf16`,
  `gpu_embedding_gather_bf16_to_f32`, `gpu_rope_half_bf16`,
  `gpu_transpose_to_heads_bf16`, `gpu_transpose_from_heads_bf16`,
  `gpu_repeat_kv_bf16`, `gpu_causal_mask_bf16` — fused kernels for
  attention building blocks, all kept GPU-resident.

- REQ-8: Numerical-prep kernel — `gpu_block_reduce_max_abs_bf16` —
  block-reduction primitive for quantisation calibration.

- REQ-9: Every `unsafe { ... }` block carries a SAFETY comment naming
  the kernel entry, the buffer-allocation site, the exclusive-mutable-
  borrow invariant, the bound-check guarantee, and the `n as u32`
  non-truncation argument (matching `R-CODE-1` leaf-primitive policy).

- REQ-10: `n == 0` is an early-return path that allocates a 0-length
  output (no kernel launch). Mirrors PyTorch's empty-tensor short
  circuit in TensorIterator (`aten/src/ATen/native/TensorIterator.cpp`).

## Architecture

### Half-precision PTX idiom

bf16 is the top 16 bits of an f32 with zero-padded low bits, so
in-register conversions are bit ops:

- **bf16 → f32**: `mov.b32 %f, {%zero16, %h}` — splat into the high
  half of a b32 register. Lossless. Pattern from NVIDIA's
  `cuda_bf16.hpp::__internal_bfloat162float` and PyTorch's
  `c10::detail::f32_from_bits` (cf. `c10/util/BFloat16.h`).

- **f32 → bf16 (round-to-nearest-even)**: add the rounding bias
  `0x7FFF + bit[16]` to the f32 bits, then `shr 16`. Same pattern as
  PyTorch's `round_to_nearest_even` in `c10/util/BFloat16.h`.

All arithmetic happens in `.f32` registers per thread; storage is
always `u16` (`.b16`) in global memory. No whole-tensor f32 intermediate
materialisation.

### Elementwise binary (REQ-1)

`pub fn gpu_mul_bf16 in bf16.rs` is the reference. Each binary kernel
reads `a[i]`, `b[i]` (one `.b16` each), splats both to f32 via the
bf16→f32 trick, computes the math in `.f32`, then encodes back with
the rounding bias. Output stored at `out[i]` as `.b16`. Bound check
`setp.ge.u32 %p, %r_tid, %n_reg; @%p bra DONE;` ensures threads past
the logical `n` skip the write. Wrapper at the Rust level uploads the
PTX once via `module_cache::get_or_compile`, allocs a `n`-element u16
output, and dispatches via `stream.launch_builder().launch(cfg)`. The
`unsafe` block carries the full SAFETY contract.

Non-test production consumer: `crate::backend_impl::CudaBackendImpl`'s
bf16 dtype arm dispatches the matching `gpu_*_bf16` function (e.g.
`fn elementwise_add_bf16` calls `crate::bf16::gpu_add_bf16`; see the
40+ call sites listed under `backend_impl.rs:5147..5444`).

### Elementwise unary (REQ-2)

`pub fn gpu_silu_bf16/gpu_relu_bf16/gpu_fatrelu_bf16/gpu_gelu_bf16 in
bf16.rs` and the simpler transcendental family
(`gpu_exp_bf16/gpu_log_bf16/gpu_tanh_bf16/gpu_sigmoid_bf16`) follow the
same decode-compute-encode pattern. `gpu_silu_bf16` computes
`x * sigmoid(x) = x / (1 + exp(-x))` via `ex2.approx.f32` on
`-x * log2(e)`. `gpu_gelu_bf16` uses the tanh-approximation polynomial.
Non-test production consumer: `backend_impl.rs:5174` (`gpu_gelu_bf16`),
`5182` (`gpu_silu_bf16`), `5190` (`gpu_relu_bf16`), `5413..5437` (the
transcendentals).

### Scalar / broadcast (REQ-3, REQ-4)

`pub fn gpu_scale_bf16 in bf16.rs` takes an `f32` scalar parameter
encoded as a PTX `.param .f32` and multiplies the buffer in place
(producing a fresh output). Non-test consumer: `backend_impl.rs:5228`,
`5511` (the f16-aliased call). The broadcast family
(`gpu_broadcast_{add,sub,mul,div}_bf16`) takes both inputs' shapes plus
the output shape, allocates the output via the backend's pool, and
dispatches a stride-aware PTX kernel. Non-test consumer:
`backend_impl.rs:5286..5340`.

### Reductions (REQ-5)

`pub fn gpu_sum_bf16/gpu_mean_bf16 in bf16.rs` are global reductions;
`gpu_sum_axis_bf16_bf16/gpu_mean_axis_bf16_bf16` are axis reductions
preserving bf16 output. Each accumulates in f32 (PyTorch parity per
`aten/src/ATen/native/cuda/Reduce.cuh::ReduceConfig`) and re-encodes
the final scalar/row sum to bf16. Non-test consumer:
`backend_impl.rs:5349..5404`.

### Normalisation + transformer primitives (REQ-6, REQ-7)

`pub fn gpu_layernorm_bf16/gpu_rmsnorm_bf16/gpu_softmax_bf16 in bf16.rs`
are composite kernels combining mean + variance/RMS + apply in a
single PTX entry. `gpu_softmax_bf16` uses the max-subtract trick for
numerical stability (matches PyTorch's
`aten/src/ATen/native/cuda/SoftMax.cu::cunn_SoftMaxForward`). Transformer
primitives (`embedding_gather`, `rope_half`, `transpose_to/from_heads`,
`repeat_kv`, `causal_mask`) are fused kernels reusing the bf16 decode/
encode PTX scaffolding. Non-test consumer for `softmax`:
`backend_impl.rs:5147`; for `layernorm`: `5165`; for `rmsnorm` (called
via the `f16` aliasing path, since both share the same `DType::BF16` →
bf16 kernel dispatch).

### SAFETY discipline (REQ-9)

Every `unsafe { stream.launch_builder(&f).arg(a)...launch(cfg)? }`
block is preceded by a multi-line SAFETY comment naming: (a) the
compile site for `f` via `module_cache::get_or_compile`, (b) the entry
point's parameter signature matching the argument push order, (c) the
input buffer length binding for `n`, (d) the fresh allocation of `out`
ensuring non-aliasing with the immutable inputs, (e) the PTX bound
check that limits reads/writes to `[0, n)`, (f) the non-truncation of
the `n as u32` cast for grid sizing. R-CODE-1 grandfathers raw CUDA
kernel launches (a leaf primitive) as `unsafe`-allowed.

### n == 0 (REQ-10)

Every public entry point opens with `if n == 0 { return
Ok(stream.alloc_zeros::<u16>(0)?); }` — no PTX launch, no PTX compile
attempt. Mirrors PyTorch's TensorIterator `iter.numel() == 0` short
circuit at `aten/src/ATen/native/TensorIterator.cpp` (the `iter.for_each`
loop simply doesn't run).

## Parity contract

`parity_ops = []` for this route (per S5: this is INFRASTRUCTURE that
underlies many parity ops — e.g. `add`, `mul`, `relu`, `softmax`,
`layernorm` parity-sweep ops each exercise this file via the dtype-bf16
arm of `backend_impl.rs`). Per-op parity is enforced one layer up at
the `ferrotorch-core` op crate; the bf16 path is the dtype-specific
implementation that the op crate dispatches through when the input
dtype is `DType::BF16`.

Edge cases preserved by these kernels:

- **NaN propagation**: f32 arithmetic in PTX preserves NaN per IEEE-754
  (`add.f32`, `mul.f32` of NaN with anything → NaN). The bf16 encode
  step preserves NaN bit patterns (the rounding-bias add does not
  rescue NaN payloads — both PyTorch and ferrotorch produce a bf16 NaN
  bit pattern). Verified through the `bf16_vit_block_e2e.rs`
  integration test and the inline `bf16::tests` unit tests in the
  file.

- **±Inf**: `exp(big) → +Inf` on f32, then `+Inf` encoded as bf16 bit
  pattern `0x7F80` (PyTorch parity).

- **Denormal**: bf16 denormals occur when the encoded exponent is 0 and
  mantissa is nonzero. The round-to-nearest-even step preserves them
  byte-for-byte with the f32→bf16 path PyTorch uses.

- **Empty input** (`n == 0`): early return without launching, allocates
  a 0-length u16 buffer.
