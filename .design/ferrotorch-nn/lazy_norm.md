## Summary

`ferrotorch-nn/src/lazy_norm.rs` implements
`LazyBatchNorm{1,2,3}d<T>` and `LazyInstanceNorm{1,2,3}d<T>` — the
deferred-`num_features`-discovery variants of the BN / IN
normalization layers mirroring
`torch.nn.LazyBatchNorm{1,2,3}d` (at
`torch/nn/modules/batchnorm.py`) and
`torch.nn.LazyInstanceNorm{1,2,3}d` (at
`torch/nn/modules/instancenorm.py`) via the `LazyModuleMixin`
protocol at `torch/nn/modules/lazy.py`. Both families discover
`num_features = input.shape()[1]` on the first forward, materialize
the inner `BatchNormNd<T>` or `InstanceNormNd<T>`, and delegate
subsequent forwards.

## Requirements

- REQ-1: `pub struct LazyBatchNorm1d<T: Float>`,
  `LazyBatchNorm2d<T>`, `LazyBatchNorm3d<T>` — each carrying `eps:
  f64`, `momentum: f64`, `affine: bool`, `inner:
  OnceLock<BatchNormNd<T>>`, and `training: AtomicBool`. Generated
  by the `lazy_batchnorm!` macro.
- REQ-2: `pub struct LazyInstanceNorm1d<T: Float>`,
  `LazyInstanceNorm2d<T>`, `LazyInstanceNorm3d<T>` — each carrying
  `eps: f64`, `affine: bool`, `inner: OnceLock<InstanceNormNd<T>>`,
  and `training: AtomicBool`. Generated by the `lazy_instancenorm!`
  macro. (`InstanceNorm` has no `momentum` field — that's the
  signature divergence from BN that motivated the separate macro.)
- REQ-3: `LazyBatchNormNd::new(eps, momentum, affine)` and
  `LazyInstanceNormNd::new(eps, affine)` — infallible constructors
  returning `Self`.
- REQ-4: `materialize(num_features)` — eagerly construct the inner
  module. Idempotent: first call wins.
- REQ-5: `<LazyNormNd as Module>::forward` validates `input.ndim()`
  matches the expected rank (via `channels_from_input`), reads
  `num_features = input.shape()[1]`, materializes if needed, then
  delegates to `inner.forward(input)`. If the OnceLock is somehow
  still empty after `materialize` succeeded (invariant violation),
  returns `FerrotorchError::Internal`.
- REQ-6: `num_features()` accessor on the BN variants — returns
  `Option<usize>` (None before materialize, `Some` after).
- REQ-7: `running_mean()` / `running_var()` accessors on the BN
  variants (added in #1072) — return `Option<Vec<f64>>` (None
  before materialize, snapshot vec of length `num_features` after).
  These mirror PyTorch's `.running_mean` / `.running_var` tensor
  attribute access on a lazy BN.
- REQ-8: `Module<T>` trait — `parameters` / `parameters_mut` /
  `named_parameters` forward through `inner` and return empty
  pre-init. `train` / `eval` cascade into `inner`.
- REQ-9: `is_initialized()` accessor.

## Architecture

### `channels_from_input` helper

`fn channels_from_input<T: Float>(input: &Tensor<T>, op: &str,
expected_ndim: usize) -> FerrotorchResult<usize>` in `lazy_norm.rs`
— same shape as the helper in `lazy_conv_transpose.rs`; the two
helpers are independent copies (deduplication is a follow-up if
called for).

### `lazy_batchnorm!` macro (REQ-1)

`macro_rules! lazy_batchnorm!($name, $inner, $expected_ndim,
$kind)` in `lazy_norm.rs`. Generates the struct, accessors,
constructor, materialize, and `Module` impl. Used to instantiate
`LazyBatchNorm1d` (with `BatchNorm1d` and ndim=2),
`LazyBatchNorm2d` (with `BatchNorm2d` and ndim=4), and
`LazyBatchNorm3d` (with `BatchNorm3d` and ndim=5). NOTE:
`LazyBatchNorm1d` also accepts 3-D inputs at the inner `BatchNorm1d`
level; the macro's `$expected_ndim` is the strictest rank for the
shape check, which means 3-D inputs are currently rejected at the
lazy layer (divergence with upstream `LazyBatchNorm1d` which
accepts both 2-D and 3-D); the inline comment in `lazy_norm.rs`
flags this with `// BatchNorm1d also accepts 3D`.

### `lazy_instancenorm!` macro (REQ-2)

`macro_rules! lazy_instancenorm!($name, $inner, $expected_ndim,
$kind)` in `lazy_norm.rs`. Same shape but with the `BatchNorm`-
specific `momentum` / `running_mean` / `running_var` plumbing
omitted. Used to instantiate `LazyInstanceNorm{1,2,3}d`.

### Forward (REQ-5)

`<LazyNormNd as Module>::forward` body in the macro:
1. `c = channels_from_input(input, $kind, $expected_ndim)?`.
2. If `inner.get().is_none()`, call `materialize(c)`.
3. `inner.get().ok_or_else(...)` — Internal error if the invariant
   "materialize populated the OnceLock" fails; otherwise delegate.

### `running_mean` / `running_var` (REQ-7)

`pub fn running_mean(&self) -> Option<Vec<f64>>` and
`pub fn running_var(&self) -> Option<Vec<f64>>` defined inside
the `lazy_batchnorm!` macro body. Snapshot via
`m.running_mean()` on the inner BatchNorm if materialized; None
otherwise.

### Trait surface (REQ-8, REQ-9)

`impl<T: Float> Module<T> for LazyNormNd<T>` in the macro. Train /
eval cascade into the inner. `is_initialized` checks
`inner.get().is_some()`.

### Non-test production consumers

- `pub use lazy_norm::{LazyBatchNorm1d, LazyBatchNorm2d,
  LazyBatchNorm3d, LazyInstanceNorm1d, LazyInstanceNorm2d,
  LazyInstanceNorm3d}` at `ferrotorch-nn/src/lib.rs`.
- Dynamic-shape vision pipelines (where the input channel count to
  a BN layer is determined by a preceding lazy conv) use these in
  conjunction with `LazyConv2d` to defer all in-channels decisions
  until first forward.

## Parity contract

`parity_ops = []`. The BN / IN parity is owned by
`.design/ferrotorch-nn/norm.md` and its `nn.functional.batch_norm`
/ `instance_norm` ops. The lazy plumbing is verified by lib tests
in this file.

Edge cases:
- **`running_mean` / `running_var` accessors** — pinned to return
  `None` pre-init and `Some(vec of length C)` post-init, with the
  initial values being zeros and ones respectively (the inner BN's
  default state). Tested by
  `lazy_batchnorm_accessors_some_after_materialize`.
- **Training-mode drift** — after one training-mode forward over
  non-zero input, `running_mean` must drift away from zero
  (discriminates against a stub returning zeros) and `running_var`
  must remain positive (discriminates against a stub returning all
  zeros or the unchanged initial 1.0 vector).
- **Wrong-rank input** — rejected by `channels_from_input`.
